## Set Tracker ‚Äî project context

### Tech stack
- **Next.js 16.2.6** with App Router, React 19, TypeScript 5
- **Supabase** ‚Äî PostgreSQL, Row-Level Security, Supabase Auth (magic link OTP only, no passwords)
- **Tailwind CSS v4** ‚Äî config is in `globals.css` via `@theme` directive; there is no `tailwind.config.ts`

- **Sonner** for toasts, **Lucide React** for icons, **Zod** for server action validation

### Critical conventions
- **Routing middleware is `src/proxy.ts`**, not `src/middleware.ts`. Next.js 16 changed this convention. Do not create a `middleware.ts` file.
- **Tailwind v4**: All theme customisation is in `src/app/globals.css` under `@theme inline {}`. There is no `tailwind.config.ts`. Do not create one.
- **@base-ui/react** is the headless primitive library (used by shadcn here). Do not import from `@radix-ui/*`.
- **Auto-generated types**: `src/lib/types/database.ts` is generated by `supabase gen types typescript`. Regenerate with the Supabase CLI after schema changes; never hand-edit.
- **`cn()` utility**: import from `@/lib/utils`, not from shadcn or elsewhere.

### Auth
Supabase SSR auth with cookie-based sessions. Magic link OTP only ‚Äî no password flow exists.
- `(auth)/login` ‚Üí user enters email ‚Üí `supabase.auth.signInWithOtp()` ‚Üí email link ‚Üí `/auth/callback`
- `src/proxy.ts` middleware redirects unauthenticated requests to `/login`
- `(app)/layout.tsx` also checks auth and redirects (double protection)
- Public routes: `/login`, `/auth/*`

### Server actions
All mutations are Next.js Server Actions (`"use server"`) in `src/lib/actions/`:
- `workout.ts` ‚Äî `getActiveSession`, `getWorkoutSession`, `createWorkoutSession`, `addSet`, `deleteSet`, `finishWorkout`, `cancelWorkout`, `addExerciseToSession`, `removeExerciseFromSession`, `getRecentWorkouts`, `getWorkoutHistory`, `getSessionDetail`, `getExerciseHistory`, `getTemplateNameById`
- `templates.ts` ‚Äî template CRUD (`getTemplates`, `getTemplateDetail`, `createTemplate`, `deleteTemplate`)
- `exercises.ts` ‚Äî `getExercises` (with optional search filter), `createExercise`

Actions call `supabase.auth.getUser()` for auth, use Zod for input validation, and call `revalidatePath()` after mutations.

### Data model (Supabase tables)
- `profiles` ‚Äî user display name (auto-created on signup via trigger)
- `exercises` ‚Äî global exercise library (seeded with 60+ exercises across 8 categories)
- `workout_sessions` ‚Äî active and completed workout sessions; `source_template_id` (nullable FK ‚Üí `workout_templates`) tracks which template started the session
- `session_exercises` ‚Äî ordered exercises within a session (`display_order`)
- `set_entries` ‚Äî individual sets (`weight_kg`, `reps`, `set_number`)
- `workout_templates` ‚Äî user-saved reusable templates
- `workout_template_exercises` ‚Äî exercises in a template

All tables have Row-Level Security enabled.

### Active workout flow
1. Dashboard calls `createWorkoutSession()` (or `TemplateActions.tsx` calls it when starting from a template) ‚Üí navigates to `/workout/active?session=<id>` and writes the ID to `localStorage` as `activeSessionId` (side-write only ‚Äî navigation is via URL param, not localStorage)
2. The dashboard's "Continue Workout" button is driven by `getActiveSession()` (server-side DB query), not by reading localStorage
3. `/workout/active?session=<id>` ‚Äî SSR page fetches session, passes to `ActiveWorkout.tsx` (client component)
4. `ActiveWorkout.tsx` handles timer, set logging (optimistic), exercise add/remove, history display
5. Set-delete and exercise-remove are optimistic with a 4s undo window ‚Äî the server call is deferred via `setTimeout` and cancelled if the user clicks Undo, so an undo never races a completed server delete
6. Finish ‚Üí `finishWorkout()` writes the template **before** setting `finished_at`, so a failed template save leaves the session active and the user can retry; Cancel ‚Üí `cancelWorkout()` ‚Äî both remove `activeSessionId` from localStorage
7. Runtime errors inside the protected `(app)` route group are caught by `src/app/(app)/error.tsx`; `src/app/(app)/workout/active/error.tsx` overrides it with a workout-specific message. Both are Next.js App Router error boundaries (must be Client Components, receive `error` + `reset` props)

### CI / Deployment
- **Vercel** ‚Äî production and preview deployments are automatic on push/PR
- **CI** (`.github/workflows/ci.yml`) ‚Äî runs typecheck + lint + build on every PR
- **Migrations** (`.github/workflows/migrate.yml`) ‚Äî automatically runs `supabase db push` against both dev and prod (two parallel jobs) when any file under `supabase/migrations/` is pushed to `main`; uses GitHub secrets `SUPABASE_DEV_DB_URL` and `SUPABASE_PROD_DB_URL`
- **Env vars**: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` (client + server); `SUPABASE_SERVICE_ROLE_KEY` (server only); `SUPABASE_DB_URL` (local Supabase CLI only ‚Äî CI uses `SUPABASE_DEV_DB_URL` / `SUPABASE_PROD_DB_URL`). See `.env.example`.

#### Vercel env var gotchas (these have bitten us repeatedly)
- **Editing a Vercel env var does NOT update existing deployments.** `NEXT_PUBLIC_*` values are **inlined into the bundle at build time**, so after changing one you must trigger a **fresh build of that branch** (push a commit / redeploy *that* branch with build cache off). "Redeploy" in the dashboard often rebuilds the wrong commit (e.g. `main`) ‚Äî confirm it targets the branch you mean.
- **Vars are scoped per environment (Production / Preview / Development).** A var set only for Production leaves Preview builds with an empty value ‚Üí the proxy middleware throws `Invalid supabaseUrl: Must be a valid HTTP or HTTPS URL` and **every preview route 500s** (build still succeeds, since the build uses placeholder values). Ensure each Supabase var is attached to **Preview** too, pointing at the **dev** project (`msahweejzdmfblpjzxrt`).
- **Key placement matters:** `NEXT_PUBLIC_SUPABASE_ANON_KEY` must be the **publishable** key (`sb_publishable_‚Ä¶`); the **secret** key (`sb_secret_‚Ä¶`) belongs only in `SUPABASE_SERVICE_ROLE_KEY` (no `NEXT_PUBLIC_` prefix). A secret key in any `NEXT_PUBLIC_*` var ships to the browser and Supabase throws **`Forbidden use of secret API key in browser`** on the first client call (e.g. login `signInWithOtp`).
- **Verify what actually shipped** without a browser: fetch a built JS chunk and grep for the key prefix, e.g. `sb_publishable_` (good) vs `sb_secret_` (wrong) ‚Äî the inlined value is in the client bundle.
